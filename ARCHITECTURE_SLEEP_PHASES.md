# Архитектура: Постепенная настройка режима сна (Phase-Based Sleep Schedule)

Документ описывает всё, что нужно реализовать. Используется как контекст для Claude Code.

> **Принятые решения** отмечены меткой [РЕШЕНО].

---

## 1. Суть фичи

Пользователь хочет перейти с текущего режима сна (например, подъём в 9:00) на целевой (подъём в 6:00). Резкий переход создаёт недосып и срыв. Приложение рассчитывает постепенный план перехода: фазы по 7 дней, сдвиг на выбранное количество минут за фазу. Все разделы приложения (протокол, распорядок, инструкция) автоматически подстраивают свои времена под текущую фазу.

Когда план завершён — фазы исчезают, приложение работает в обычном режиме с целевыми временами.

## 2. Модель данных

### 2.1 SleepPlan — главный объект плана

```typescript
interface SleepPlan {
  id: string;                    // уникальный ID плана
  status: 'active' | 'completed' | 'archived';
  
  // Входные данные от пользователя (визард)
  currentWake: string;           // "09:00" — текущий подъём
  targetWake: string;            // "06:00" — целевой подъём
  desiredSleepHours: number;     // 8.5 — желаемая длительность сна в часах
  
  // [РЕШЕНО] stepMinutes настраивается пользователем (15/20/30/45/60)
  stepMinutes: number;           // шаг сдвига в минутах, выбирается в визарде
  phaseDays: number;             // 7 — длительность фазы (всегда 7, не настраивается)
  startDate: string;             // "2026-06-01" — дата начала (ISO)
  
  phases: Phase[];               // массив фаз, рассчитывается при создании
  createdAt: string;             // ISO datetime
}
```

> [РЕШЕНО] Поле `currentSleep` убрано — оно не используется ни в одном расчёте.

### 2.2 Phase — одна фаза плана

```typescript
interface Phase {
  number: number;                // 1, 2, 3... (нумерация с 1)
  wake: string;                  // "08:30" — подъём этой фазы
  bed: string;                   // "00:00" — отбой этой фазы
  startDate: string;             // "2026-06-01" — первый день фазы (ISO date)
  endDate: string;               // "2026-06-07" — последний день фазы (ISO date)
  color: string;                 // цвет фазы для UI (hex)
}
```

### 2.3 RoutineStep — шаг вечернего распорядка

```typescript
interface RoutineStep {
  id: string;                    // уникальный ID
  name: string;                  // "Душ + зубы"
  emoji: string;                 // "🚿"
  offsetMinutes: number;         // -50 (отрицательное = до отбоя, 0 = отбой)
  order: number;                 // порядок сортировки (0, 1, 2...)
  isFixed: boolean;              // true только для "В кровать" (нельзя удалить)
}
```

Шаги хранятся как шаблон, не зависящий от плана. Конкретные времена рассчитываются при рендере на основе `bed` текущей фазы.

### 2.4 DayContext — контекст конкретного дня

Не хранится, рассчитывается на лету. Используется при рендере любого раздела.

```typescript
interface DayContext {
  date: string;                  // "2026-06-05"
  hasPlan: boolean;              // есть ли активный план
  phase: Phase | null;           // текущая фаза (или null)
  wake: string;                  // целевой подъём на сегодня
  bed: string;                   // целевой отбой на сегодня
  
  // Рассчитанные времена протокола
  protocol: {
    caffeineUntil: string;       // wake + 6ч
    trainingUntil: string;       // bed − 4ч
    screensOff: string;          // bed − 2ч
    lastMeal: string;            // bed − 2ч 20мин
    noExercise: string;          // bed − 2ч
    shower: string;              // bed − 75мин
    toilet: string;              // bed − 10мин
  };
  
  // Шаги распорядка с рассчитанными временами
  routine: {
    step: RoutineStep;
    time: string;                // конкретное время ("20:30")
  }[];
}
```

### 2.5 Прогресс распорядка

> [РЕШЕНО] Хранится как объект на день (аналогично `entry.protocol`), а не составным ключом.

```typescript
// В хранилище routineProgress, keyPath: 'date'
interface RoutineProgressEntry {
  date: string;                  // "2026-06-05"
  steps: {                       // объект: stepId → boolean
    [stepId: string]: boolean;   // { step_1: true, step_3: true }
  };
}
```

## 3. Движок фаз: алгоритм расчёта

### 3.1 Генерация фаз при создании плана

```
Вход:
  currentWake = "09:00"
  targetWake = "06:00"
  desiredSleepHours = 8.5
  stepMinutes = 30            // [РЕШЕНО] выбирается пользователем из 15/20/30/45/60
  phaseDays = 7
  startDate = "2026-06-01"

Алгоритм:
  1. diffMinutes = currentWake - targetWake  // в минутах: 540 - 360 = 180
  2. phaseCount = ceil(diffMinutes / stepMinutes)  // ceil(180 / 30) = 6
  3. Для каждой фазы i (от 1 до phaseCount):
     - wake = currentWake - (i * stepMinutes)
       фаза 1: 09:00 - 30 = 08:30
       фаза 2: 09:00 - 60 = 08:00
       ...
       фаза 6: 09:00 - 180 = 06:00  ← целевой
     - Если wake < targetWake → wake = targetWake
     - bed = wake - (desiredSleepHours * 60)
     - startDate = planStartDate + ((i - 1) * phaseDays)
     - endDate = startDate + phaseDays - 1
     - color = из палитры по индексу

  4. Проверка: последняя фаза.wake === targetWake

Выход: массив Phase[]
```

### 3.2 Палитра цветов для фаз

Используется фиксированная палитра. Если фаз больше, чем цветов — цвета циклятся.

```
Палитра (до 8 фаз):
  Фаза 1: #EF4444  (красный — далеко от цели)
  Фаза 2: #F97316  (оранжевый)
  Фаза 3: #EAB308  (жёлтый)
  Фаза 4: #84CC16  (лайм)
  Фаза 5: #22C55E  (зелёный)
  Фаза 6: #10B981  (изумрудный — цель)
  Фаза 7: #06B6D4  (если нужны доп. фазы)
  Фаза 8: #8B5CF6
```

Логика: от красного к зелёному — визуально показывает прогресс к цели.

### 3.3 Определение DayContext для конкретной даты

```
getDayContext(plan, routineSteps, dateStr):
  1. Если plan === null || plan.status !== 'active':
     → вернуть { hasPlan: false, wake: plan?.targetWake || "06:00", ... }
  
  2. Если дата до startDate:
     → вернуть { hasPlan: true, phase: null, wake: currentWake, ... }
  
  3. Найти фазу: plan.phases.find(p => date >= p.startDate && date <= p.endDate)
  
  4. Если дата за пределами всех фаз (после последней):
     → вернуть { hasPlan: true, phase: null, wake: targetWake, bed: рассчитанный }
  
  5. Рассчитать протокольные времена (см. раздел 3.4)
  6. Рассчитать времена распорядка (см. раздел 3.5)
  7. Вернуть DayContext
```

### 3.4 Формулы протокольных времён

Все времена рассчитываются от `wake` и `bed` текущей фазы.

```
caffeineUntil   = wake + 360мин       // +6 часов
trainingUntil   = bed  - 240мин       // −4 часа
screensOff      = bed  - 120мин       // −2 часа
lastMeal        = bed  - 140мин       // −2 часа 20 минут
noExercise      = bed  - 120мин       // −2 часа
shower          = bed  - 75мин        // −1 час 15 минут
toilet          = bed  - 10мин        // −10 минут
```

### 3.5 Расчёт времён распорядка

```
Для каждого RoutineStep:
  time = bed + offsetMinutes

Пример: bed = 23:00, step.offsetMinutes = -50
  time = 23:00 - 50мин = 22:10
```

Шаги сортируются по `offsetMinutes` по возрастанию (от самого раннего к bed).

## 4. Визард «Настроить режим»

### 4.1 Точки входа

> [РЕШЕНО] Визард запускается только из настроек.

- Кнопка «Новый план» в настройках → ввод подтверждения "новый план" → визард
- Если есть активный план, он архивируется при создании нового

### 4.2 Шаги визарда (6 шагов)

> [РЕШЕНО] Шаг `currentSleep` убран. Добавлен шаг выбора `stepMinutes`.

**Шаг 1/6: «Во сколько ты обычно встаёшь?»**
- Ввод: time picker, шаг 15 минут
- Значение по умолчанию: 09:00
- Сохраняется в `currentWake`

**Шаг 2/6: «Во сколько хочешь вставать?»**
- Ввод: time picker, шаг 15 минут
- Значение по умолчанию: 06:00
- Сохраняется в `targetWake`

**Шаг 3/6: «На сколько сдвигать подъём за неделю?»**
> [РЕШЕНО] Настраиваемый шаг (15/20/30/45/60 минут), без дефолта.

- Ввод: предзаданные кнопки `(15) (20) (30) (45) (60)` мин
- Без дефолта — пользователь обязан выбрать, кнопка «Далее» неактивна
- Живой превью под кнопками, пересчитывается мгновенно при выборе:
  ```
  6 фаз · 42 дня
  Финиш: 12 июля
  ```
- Сохраняется в `stepMinutes`

**Шаг 4/6: «Сколько часов сна тебе нужно?»**
- Ввод: выбор из вариантов: 7 / 7.5 / 8 / 8.5 / 9
- Без дефолта
- Подсказка: «Большинству взрослых нужно 7.5–9 часов»
- Сохраняется в `desiredSleepHours`

**Шаг 5/6: «Настрой свой вечерний распорядок»**
- Экран: редактор шагов распорядка (подробно в разделе 5.4)
- Предзаполнен дефолтным набором шагов (раздел 4.3)
- Внизу: живой превью с временами по первой фазе

**Шаг 6/6: «Твой план» — экран-превью**
- Показывает количество фаз, дней, даты старта и финиша
- Мини-таблица фаз с цветными точками
- Кнопка: «Начать» → создаёт план, переходит на главный экран

### 4.3 Дефолтный набор шагов распорядка

```
Порядок  Эмодзи  Название                   Смещение
  0       🍽     Подготовка к ужину          -160 мин (2ч 40мин)
  1       📴     Выключить экраны            -120 мин (2ч)
  2       🥗     Ужин                        -110 мин (1ч 50мин)
  3       🚶     Отдых / прогулка            -70 мин  (1ч 10мин)
  4       🚿     Душ + зубы                  -50 мин
  5       🛋     Расслабиться, походить      -15 мин
  6       🛏     В кровать                   0 мин    (нельзя удалить)
```

### 4.4 Валидация визарда

```
- targetWake >= currentWake → ошибка: «Целевой подъём должен быть раньше текущего»
- targetWake === currentWake → сообщение: «Ты уже на целевом режиме!»
- bed получается раньше 20:00 или позже 03:00 → предупреждение (не блок)
- stepMinutes не выбран → кнопка «Далее» неактивна
- Минимум 1 шаг в распорядке (кроме фиксированного «В кровать»)
```

## 5. Изменения в каждом разделе приложения

### 5.1 Навигация (шапка)

> [РЕШЕНО] Версия приложения убрана из шапки. Вместо неё — иконка ⚙️, открывающая выезжающую панель настроек.

**Текущее состояние:**
```
  [v43]  Sleep Tracker            [🌙]
       ← 29 мая, четверг →     [Сегодня]
```

**Новое состояние:**
```
  [⚙️]  Sleep Tracker            [🌙]
  [===■====-------]              ← прогрессбар фаз (если есть план)
       ← 29 мая, четверг →     [Сегодня]
  Фаза 2 · подъём 8:00 · отбой 23:30
```

Прогрессбар:
- Горизонтальная полоса, разделённая на сегменты (по одному на фазу)
- Каждый сегмент окрашен цветом фазы
- Точка/маркер ■ на текущем дне
- Пройденная часть — яркая, будущая — полупрозрачная (opacity 0.35)

Тап по прогрессбару → раскрывается мини-календарь с раскрашенными днями.

Подпись под датой:
- Формат: `Фаза N · подъём HH:MM · отбой HH:MM`
- Цветная точка слева = цвет фазы
- Если дата вне плана: подпись не показывается

**Без активного плана:**
Прогрессбар и подпись не показываются. Навигация как раньше (без версии).

### 5.2 Вкладка «Трекер»

**Структура полей — без изменений.**

**Добавляется:**

1. **Плашка фазы** — сверху формы, перед полями
2. **Индикатор попадания** — рядом с полями «встал» и «лёг»
   - |факт − цель| ≤ 15 мин → ✅
   - |факт − цель| > 15 мин → мягкая оранжевая подсветка (не красная, не тревожная)
3. **Подсказка** — мелким текстом: «Цель фазы: 07:30»

### 5.3 Вкладка «Протокол»

**Структура — без изменений.** Все времена подставляются из DayContext.

Подсказки (тексты «зачем это нужно») — не зависят от фазы, остаются как есть.

### 5.4 Вкладка «Распорядок»

**Полностью переработана.** Рендерит пользовательские шаги с рассчитанными временами + чекбоксы выполнения.

Кнопка ✏️ → открывает редактор шагов.

### 5.5 Вкладка «Инструкция»

> [РЕШЕНО] Даты «Вечер 20», «Утро 21» — статичный пример конкретной пары дней, не становятся динамическими.

Текст остаётся как есть. Не содержит хардкоженных часов, привязанных к протоколу.

### 5.6 Вкладка «История»

Каждая запись получает цветную метку фазы. При раскрытии — строка «цель vs факт».

Записи от завершённых/архивных планов сохраняют свои метки навсегда.

## 6. Жизненный цикл плана

### 6.1 Состояния

```
[Нет плана] ──→ Настройки → «Новый план» → ввод "новый план" → Визард ──→ [Plan: active]
                                                                              │
                                                                      (дни идут, фазы проходят)
                                                                              │
                                                                  последний день последней фазы
                                                                              │
                                                                              ▼
                                                                      [Plan: completed]
                                                                              │
                                                                  (фазы исчезают из UI,
                                                                   времена фиксируются
                                                                   на целевых значениях)

В любой момент при active плане:
  Настройки → «Новый план» → ввод "новый план" → [Plan: archived] → визард
  Настройки → «Сбросить план» → подтверждение → [Plan: archived] → [Нет плана]
```

### 6.2 Переход между состояниями

**active → completed:** автоматически при `today > lastPhase.endDate`. Поздравление один раз.

**active → archived:** вручную через «Новый план» или «Сбросить план» в настройках.

### 6.3 Режим без активного плана

- Протокол, распорядок, инструкция используют целевые времена из последнего плана
- Если планов не было — дефолты (wake 06:00, bed 21:30)
- Прогрессбар, плашки, метки — скрыты

## 7. Граничные случаи

### 7.1 Отбой переходит через полночь

Нормально. Отображаем как «00:30» с пометкой «(следующий день)». Уже есть в приложении.

### 7.2 Нечётное количество минут

Последняя фаза всегда = targetWake. Предпоследняя фаза может иметь шаг меньше выбранного.

### 7.3 Пользователь не заполняет трекер несколько дней

Фазы идут по календарю, не по заполненности. Не создавать давление.

### 7.4 Просмотр будущего дня

Показываем фазу с целевыми временами. Поля трекера пусты.

### 7.5 Дата до начала плана

Обычный вид без фазы.

### 7.6 currentWake и targetWake совпадают

Визард: «Ты уже на целевом режиме!». Не создаёт план.

### 7.7 Разница < stepMinutes

Создаётся одна фаза на 7 дней с targetWake для закрепления.
