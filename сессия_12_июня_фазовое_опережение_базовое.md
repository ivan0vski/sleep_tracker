# Сессия 12 июня 2026 — Базовое фазовое опережение

## Цель сессии

Сделать фазовое опережение в sleep tracker научно обоснованным. Старая модель была примитивной: фаза всегда 7 дней, сдвиг X минут за неделю. Это не соответствует научным данным — оптимальная скорость циркадного сдвига ~30 мин/день (Burgess 2003, Eastman 2005). Чтобы это поддержать, нужно дать пользователю выбирать длину фазы от 1 до 7 дней.

---

## Научная основа (из документа «базовая версия фазового опережения для себя.md»)

### Ключевые исследования

1. **Burgess et al. (2003)** — реальный циркадный сдвиг без мелатонина, только с утренним светом — ~0.5 часа/сутки. С мелатонином до 0.8 ч/сутки.

2. **Eastman et al. (2005)** — смещение на 2 часа/день не лучше чем на 1 час/день. Слишком быстро для адаптации — появляются задержки засыпания, снижается эффективность сна.

3. **Burgess (2010)** — частичная депривация сна значительно уменьшает фазовое опережение в ответ на утренний свет. Недосып делает свет менее эффективным.

4. **Burgess (2013)** — длина непрерывного тёмного блока перед утренним светом критически важна. Длинная ночь (9ч темноты) → сдвиг 1.7ч. Короткая ночь (6ч) → только 0.7ч, даже при одинаковом общем сне. Вечерний свет противодействует утреннему сдвигу.

### Практический вывод

- **Шаг 30 мин/день** — оптимум, совпадает с реальной скоростью DLMO-сдвига
- **Подъём и отбой двигаются синхронно** — нельзя двигать подъём быстрее, чем отбой, иначе недосып съедает эффект
- **Длинная тёмная ночь** — ложиться рано не просто «чтобы выспаться», а чтобы создать длинный тёмный блок, усиливающий чувствительность к утреннему свету

### Как это связано с изменениями

Старая модель: phaseDays = 7 (всегда). Сдвиг 30 мин за 7 дней = ~4 мин/день — слишком медленно и не соответствует науке.

Новая модель: phaseDays = 1-7 (выбирает пользователь). При phaseDays=1 и step=30 получается ежедневный сдвиг 30 мин/день — точно научный оптимум.

### 3. app.js — адаптивный phase-bar с пагинацией

**Файл**: `js/app.js`

#### Проблема

При phaseDays=7 phase-bar показывает 7 квадратиков одной фазы — это нормально. Но при phaseDays=1 одна фаза = 1 квадрат, что бесполезно. При phaseDays=2 — 2 квадрата, тоже мало информации.

#### Решение: два режима отображения

**renderPhaseBar()** — теперь диспетчер:
```javascript
const planPhaseDays = activePlan.phaseDays || 7;
if (planPhaseDays <= 2) {
    renderPhaseBarPaginated(wrap, phase);
} else {
    renderPhaseBarSingle(wrap, phase);
}
```

**renderPhaseBarSingle()** — старое поведение (для phaseDays >= 3). Показывает только текущую фазу (3-7 квадратов одного цвета).

**renderPhaseBarPaginated()** — новый режим (для phaseDays <= 2). Показывает ВСЕ дни плана, разбитые на страницы по 7 штук:

1. Собирает плоский массив всех дней плана (дата + фаза + цвет)
2. Находит индекс текущей даты → вычисляет номер страницы
3. Показывает 7 ячеек текущей страницы (каждая в цвете своей фазы)
4. Если страниц > 1 — точки-индикаторы под phase-bar

**Пример**: 6 фаз по 1 дню = 6 квадратов разных цветов, одна страница, без точек.

**Пример**: 4 фазы по 2 дня = 8 квадратов (4 цвета, каждый на 2 квадрата). Страница 1: 7 квадратов, страница 2: 1 квадрат. 2 точки-индикатора.

## Архитектурные решения

### Почему phaseDays, а не "скорость сдвига"

Можно было бы ввести параметр "минут/день" вместо "дней на фазу". Но phaseDays лучше:
- Уже используется в phaseEngine.js (calculatePhases принимает phaseDays)
- Интуитивнее для пользователя: "1 фаза = 1 день" понятнее чем "30 мин/день"
- Сохраняет существующую модель: фаза = блок дней с одинаковым расписанием
- Обратно совместимо без миграции

### Почему два режима phase-bar

При phaseDays >= 3 одна фаза = 3-7 дней → достаточно квадратов чтобы показать прогресс внутри фазы. Стандартный режим работает отлично.

При phaseDays <= 2 одна фаза = 1-2 дня → 1-2 квадрата бесполезны. Нужно показать весь план целиком. Но если фаз много (например 6 фаз по 2 дня = 12 квадратов), всё не влезет → пагинация по 7 штук с точками-индикаторами.

Порог 7 квадратов на страницу — чтобы при phaseDays=7 (максимум) все квадраты одной фазы влезали на одну страницу.

### Почему превью шага 3 упрощён

На шаге 3 пользователь выбирает stepMinutes. Количество фаз зависит только от (currentWake - targetWake) / stepMinutes — phaseDays на него не влияет. Поэтому на шаге 3 показываем только "N фаз".

А вот количество дней и дата финиша зависят от phaseDays, который выбирается на шаге 4. Поэтому полный превью "N фаз · N дней · финиш" появляется только на шаге 4.
